Design spec

Talk to me — a voice AI of Noah

2026-05-20 · branch talk-to-me · scope: play-around, not production-polish

Goal

Add a Talk to me button to the homepage. Clicking it opens a real-time voice conversation with an AI version of Noah — input via microphone, output in a cloned version of Noah's voice. Every conversation is transcribed and stored so Noah can review what people asked.

This is a personal experiment, not a polished product. The bar is "interesting to play with," not "ready for thousands of users."

Decisions baked in (from brainstorm)

QuestionChoice
Conversation modeVoice both ways, real-time
Voice cloningElevenLabs, starting with Instant Voice Clone (~1 min sample)
"Trained on my data"System prompt + injected content (blog posts, bio). No fine-tuning, no RAG.
Speech-to-textStreaming STT for low latency (Deepgram)
Pipeline orchestrationDIY using Pipecat (open-source Python voice-agent framework)
HostingAstro site stays on Netlify; voice pipeline is a separate Python service on Fly.io
LogsNetlify Blobs (no new DB), surfaced via a private admin page
AccessOpen to anyone, with hard cost guardrails (session length, daily $ cap, per-IP limit)

Two deploys. The static site stays on Netlify with no change to its build. The voice pipeline lives in its own folder (voice-agent/) and deploys independently to Fly.io. The website embeds a small JS client that opens a secure WebSocket to the Fly.io service.

Components

1. Homepage button + voice UI claude builds

  • New button on src/pages/index.astro styled in the existing aesthetic (oxblood accent on hover, not a generic CTA).
  • Clicking opens an inline panel (not a modal — keep the page layout calm) with:
    • A pulsing dot indicating who's speaking (Noah / you / idle).
    • A live, scrolling transcript.
    • An End button.
    • A countdown showing remaining session time (gentle reminder that there's a cap).
  • Client-side JS uses the Pipecat web client SDK to handle mic capture, audio playback, and the WebSocket. No hand-rolled audio plumbing.
  • Posts a session start event to Netlify Function for rate-limit check before connecting.

2. Voice agent service claude builds

A small Python app in voice-agent/, deployed to Fly.io. Pipecat pipeline:

WebSocketTransport
  → Deepgram STT (streaming, with VAD/turn detection)
  → Anthropic Claude (sonnet-4-6 by default; opus-4-7 optional)
      system_prompt = persona + injected blog posts + bio
  → ElevenLabs TTS (streaming, voice_id = your cloned voice)
  → WebSocketTransport (audio back to browser)

Side effects on each session:

  • On turn end: append {role, text, ts} to an in-memory session transcript.
  • On session end (or hard cap timeout): POST the full transcript + duration + cost estimate to the Netlify logging endpoint.

The system prompt is assembled at startup by reading the markdown files in src/content/blog/, the legacy posts, and a hand-written persona.md. Cheap to rebuild on deploy.

3. Cost guardrails claude builds

  • Per-session hard cap: 5 minutes. Service emits a polite voice line ("we're hitting the time limit, talk to you next time") and disconnects.
  • Per-IP cap: 2 sessions per day. Counter in an Upstash Redis free tier (or a JSON file on a Fly.io volume — pick at impl time).
  • Site-wide daily budget: configurable env var (default $5/day). When tripped, the button on the homepage flips to "taking a break, try tomorrow". Budget is estimated from ElevenLabs chars + Deepgram minutes + Anthropic tokens, summed in the logging endpoint.
  • Profanity / abuse: out of scope for v0. If it becomes a problem we add an Anthropic content-filter pass.

4. Logging claude builds

  • New Netlify Function: netlify/functions/log-conversation.ts.
  • Accepts a POST from the voice service (shared-secret header for auth).
  • Writes a JSON blob to Netlify Blobs: conversations/<YYYY-MM-DD>/<session_id>.json.
  • Blob shape: 
    {
  "session_id": "abc123",
  "started_at": "2026-05-20T18:32:04Z",
  "duration_s": 142,
  "ip_hash": "sha256:…",
  "transcript": [
    { "role": "user", "ts": 0.0, "text": "what do you do?" },
    { "role": "noah", "ts": 1.4, "text": "I work on..." }
  ],
  "cost_estimate_usd": 0.18,
  "ended_by": "user" | "time_cap" | "budget_cap" | "error"
}

5. Private review page claude builds

  • New Astro page: /talk-to-me-logs (not linked from anywhere).
  • Gated by Netlify Identity or a simple shared-secret in the URL (e.g., ?key=… matched against an env var). Identity is overkill for a personal admin page; the shared-secret is fine.
  • Lists conversations newest-first with date, duration, first user line as a preview. Click → full transcript view.

Sequence of one conversation

1. User clicks "Talk to me"
   → browser POSTs /api/check-rate-limit
   → 200 OK { ok: true, session_token }   (or 429 with a message)

2. Browser opens WSS to voice-agent.fly.dev with session_token

3. Pipecat starts pipeline. First utterance: a short greeting in your voice.

4. Loop:
   a. user speaks → Deepgram streams interim + final transcripts
   b. on final → Claude generates streaming response
   c. tokens stream to ElevenLabs → audio chunks stream back
   d. transcripts appended to session log
   Repeat until 5-min cap or user clicks End.

5. On end:
   - service POSTs the full transcript blob to Netlify
   - service closes the WebSocket
   - browser shows "thanks, transcript saved"

What you (Noah) need to do

  1. you Create accounts and grab API keys for: ElevenLabs, Deepgram, Anthropic (you may already have this), Fly.io.
  2. you Record a ~1 minute voice sample (clean audio, quiet room, your normal speaking voice). Upload to ElevenLabs as an Instant Voice Clone. Save the voice_id.
  3. you Write the persona.md first-person bio. (Or let Claude draft a starting version from your blog posts and edit it.)
  4. you Install flyctl locally and run fly auth login.
  5. you Add the API keys as Fly.io secrets (fly secrets set …) once the service exists.
  6. you Add the matching Netlify env vars: shared secret for log POSTs, admin key for the review page, daily budget cap.
  7. you Decide a starting daily $ budget (suggested: $5/day).

Out of scope for v0

  • Mobile-optimized UI. Desktop browser only at first.
  • Cross-session memory. Each conversation starts fresh.
  • Content moderation. We're trusting the audience for v0.
  • Multi-language. English only.
  • Replay of audio in the admin page. Transcripts only — we don't store the synthesized audio.
  • RAG / fine-tuning. The corpus is small enough that injecting it into the system prompt is fine.

Known risks & open questions

  • Latency. DIY pipeline gets us ~1–2s between end-of-user-speech and start-of-AI-voice with streaming everywhere. If that feels too slow we revisit (e.g., use a managed platform like Vapi as a swap-in).
  • Voice quality. Instant Voice Clone is OK but not amazing. If the result is uncanny in a bad way, plan B is recording 30 min for the Professional clone.
  • Fly.io cold starts. The first session of the day may have a 5–10s delay if the machine has scaled to zero. We can pin it to "always on" (~$2/mo) or accept the cold start.
  • Logs as durable record. Anything anyone says to it is captured. Add a disclaimer below the button: "Heads up: conversations are transcribed and saved so I can review them."
  • Persona drift. If the system prompt isn't tight, the AI will hallucinate biographical details. Pinning the persona doc and including an explicit "if you don't know, say you don't know" rule is important.

Success criteria for v0

  • Click button → speak → hear a reply in your cloned voice, end-to-end, in under 4 seconds of latency on the first reply.
  • End the session → see the full transcript in /talk-to-me-logs?key=… within a few seconds.
  • Trigger the 5-minute cap → service ends gracefully with a voice line.
  • Run for a week without bill surprise.
